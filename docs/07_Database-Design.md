> **Version:** 1.0

# Purpose

This document defines the complete Firestore database schema for ResQ AI.

It includes:

* Collections
* Document structure
* Relationships
* Security principles
* Read/write patterns
* AI-owned collections
* Cloud Function triggers

The schema is optimized for:

* Fast dashboard loading
* AI reasoning
* Explainability
* Scalability
* Event-driven architecture

---

# Design Principles

The database follows these principles:

* User-owned data is immutable unless modified by the user.
* AI-generated insights are stored separately.
* Every AI decision is explainable.
* Collections remain shallow.
* Documents stay below Firestore limits.
* Denormalize where necessary for performance.

# Collection: users

Stores user profile information.

```json
{
  "uid": "user_123",
  "name": "John Doe",
  "email": "john@example.com",
  "photoUrl": "",
  "role": "Student",
  "timezone": "Asia/Kolkata",
  "workingHours": {
    "start": "09:00",
    "end": "18:00"
  },
  "createdAt": "...",
  "updatedAt": "..."
}
```

# Security Rules

Users can:

* Read their own documents
* Write their own tasks
* Update their preferences

Users cannot:

* Read another user's data
* Modify AI logs
* Modify risk calculations directly

Cloud Functions have elevated privileges for AI-generated collections.

---

# Data Lifecycle

Tasks

Retained indefinitely unless deleted.

Notifications

Expire after 30 days.

Agent Logs

Expire after 90 days.

Reflections

Retained permanently.

# Database Philosophy

Separate user intent from AI intelligence.

The user owns:

* Tasks
* Preferences
* Goals

The AI owns:

* Plans
* Predictions
* Recommendations
* Schedules
* Insights

This separation keeps the system modular, explainable, and easy to evolve as the AI becomes more capable.
